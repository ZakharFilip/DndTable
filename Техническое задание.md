Пример разработки технического задания

## Введение

Настоящее техническое задание распространяется на разработку веб‑ориентированной автоматизированной информационной системы **DnDTable (MVP)** — виртуального стола (2D Virtual Tabletop) для проведения настольных ролевых игр.  
Система предназначена для совместной работы группы пользователей (ведущий/GM и игроки) в рамках **партии**, ведения **сцен** и **объектов** на сцене (токены/фигуры/заметки/изображения), а также синхронного обмена изменениями между клиентами в реальном времени.

Цель создания системы:
- сократить трудозатраты на организацию “стола” и ведение игровой сцены (карта, токены, заметки);
- обеспечить совместную работу нескольких пользователей над одной сценой;
- обеспечить сохранность данных (партии/сцены/объекты/ассеты) и контроль доступа (ACL).

Работа выполняется в рамках разработки MVP проекта **DnDTable** (монорепозиторий backend+frontend+shared).

## 2 Назначение разработки

Система **DnDTable** предназначена для:
- регистрации и аутентификации пользователей (JWT, доступ к защищённым ресурсам);
- создания и ведения партий (участники, роли GM/Player, приглашения, команды/teams);
- создания сцен и управления объектами сцены через компонентную модель (ECS‑подход для операций);
- синхронного взаимодействия нескольких клиентов через **Socket.IO** (joinParty/joinScene/applyOperation/opApplied);
- загрузки и использования ассетов (изображения) в объектах сцены (Shape.kind = image);
- разграничения прав доступа на уровне действий и данных (ACL для party/scene/object/asset).

Пользователи системы (на MVP):
- **GM (ведущий)**: создание/управление партией и сценами, управление доступом, полные права редактирования;
- **Player (игрок)**: доступ и действия в рамках выданных прав (например, редактирование разрешённых объектов).

## 3 Требования к программе

### 3.1 Требования к функциональным характеристикам

Система должна обеспечивать выполнение следующих функций (вертикали MVP):

#### 3.1.1 Авторизация (Auth, BE+FE)
- Регистрация пользователя: `POST /auth/register`
- Вход пользователя: `POST /auth/login`
- Обновление токенов: `POST /auth/refresh`
- Выход: `POST /auth/logout`
- Защита роутов/ресурсов через middleware `requireAuth` (сервер) и guard маршрутов (клиент).

Критерий готовности:
- пользователь может зарегистрироваться и войти;
- защищённые запросы требуют JWT;
- refresh‑механизм обновляет токены прозрачно.

#### 3.1.2 Пользователь и друзья (Users/Friends, BE+FE)
- Профиль: `GET /users/me`, `PATCH /users/me`
- Друзья: `GET /friends`
- Запрос дружбы: `POST /friends/request { userId }`
- Принятие: `POST /friends/accept { userId }`
- Удаление: `DELETE /friends/:userId`

Критерий готовности:
- можно отправить запрос, принять, удалить друга;
- профиль редактируется.

#### 3.1.3 Партии (Parties, BE+FE)
Функции:
- CRUD партий: `/parties`
- управление участниками: `/parties/:id/members`
- приглашения: `/parties/:id/invites` (инвайты по email/пользователю)
- команды/teams: `/parties/:id/teams` (возможна вложенность)

Критерий готовности:
- можно создать партию, пригласить, принять инвайт;
- участники и команды сохраняются.

#### 3.1.4 Сцены и объекты (Scenes & Objects, BE+FE)
Функции:
- CRUD сцен: `CRUD /parties/:partyId/scenes`
- Объекты сцены: `/scenes/:sceneId/objects` (create/update/delete/move/batch)
- Компонентная модель данных объектов (через общие Zod‑схемы `@dnd-table/shared`):
  - Transform (position/rotation/scale)
  - Shape (rect/circle/image + resourceId)
  - Description (text)
  - Visibility (visibleTo/hidden)
  - Grid (enabled/size)

Критерий готовности:
- CRUD сцен и объектов работает;
- данные валидируются и сохраняются;
- объект создаётся/редактируется/рисуется на сцене (минимальный рендер на MVP).

#### 3.1.5 Realtime (BE+FE)
Функции realtime‑взаимодействия через Socket.IO:
- `joinParty { partyId }`
- `joinScene { partyId, sceneId }`
- `applyOperation { opId, actorId, targetId, type, payload, version, ... }`
- `opApplied` (broadcast всем участникам сцены)

Требования:
- авторизация сокета по JWT (на MVP допускается упрощённая реализация, но должна быть предусмотрена в архитектуре);
- валидация операций и проверка прав (ACL) на сервере;
- идемпотентность по `opId` (повторное применение не должно дублировать эффект);
- хранение/применение операций единообразно (ECS).

Критерий готовности:
- два клиента видят синхронные изменения на одной сцене.

#### 3.1.6 ACL (права, BE+FE)
Требования:
- единый механизм проверки прав для REST и Socket.IO;
- действия в рамках домена: `party.*`, `scene.*`, `object.*`, `asset.*`;
- поддержка ролей GM/Player, команд (teams) и оверрайдов.

Критерий готовности:
- действия ограничены правами, UI позволяет менять базовые настройки (минимально для MVP).

#### 3.1.7 Assets (файлы, BE+FE)
Функции:
- загрузка ассетов (multipart): `POST /assets`
- выдача ассетов: `GET /assets/:id`
Требования:
- локальное хранилище через `ASSETS_DIR` (на MVP);
- хранение метаданных, привязка к party/scene, проверка ACL;
- возможность использовать ассет в `Shape.kind=image` через `resourceId`.

Критерий готовности:
- можно загрузить файл и использовать его в объекте сцены.

---

### 3.1.8 Нормативно‑справочная информация (справочники/объекты данных)

Система оперирует следующими сущностями (минимально для MVP):
- User (пользователь): email, имя/username, avatar;
- Party: партия (название/описание/настройки);
- Member: участник партии (userId, role, teamId?);
- Invite: приглашение в партию (email/userId, статус, роль/права);
- Team: команда (возможна иерархия);
- Scene: сцена партии;
- SceneObject: объект сцены (иерархия parent/child, порядок order, components);
- Component: компонент объекта (Transform/Shape/Description/Visibility/Grid);
- Asset: ассет (файл) и метаданные;
- Operation: операция изменения (для realtime/ECS);
- ACL/Permissions: матрица прав и правила наследования/оверрайдов.

---

### 3.2 Требования к надежности

Система должна обеспечивать:
- устойчивость к сбоям клиента/сети: повторное подключение к Socket.IO, повторная синхронизация (минимально на MVP);
- защиту данных от несанкционированного доступа: JWT‑авторизация, ограничение прав через ACL;
- контроль корректности входных данных: валидация DTO (Zod‑схемы `@dnd-table/shared`), единый обработчик ошибок;
- резервное сохранение/восстановление данных:
  - на MVP: возможность создания резервной копии MongoDB (операционный процесс);
  - архитектурно: предусмотреть точки расширения для бэкапов и миграций.

Дополнительно (рекомендовано):
- ограничение частоты запросов к `/auth/*` и критическим операциям (rate limit);
- логирование ошибок и действий (аудит — позже).

### 3.3 Требования к составу и параметрам технических средств

Минимальные требования для локальной разработки/запуска:
- ОС: Windows 10/11 / Linux / macOS
- Node.js: 18+ (LTS рекомендовано)
- MongoDB: локально или Docker (Mongo 6)
- RAM: от 4 ГБ (рекомендовано 8+ ГБ)
- Диск: от 1 ГБ свободного пространства (без учёта node_modules/ассетов)
- Браузер: Chrome/Edge/Firefox актуальных версий

---

### 3.4 Требования к информационной и программной совместимости

Технологический стек (MVP):
- Backend: Node.js + TypeScript, Express (REST), Socket.IO (realtime), MongoDB (Mongoose), Zod, JWT, CORS, dotenv.
- Frontend: React + TypeScript, Vite, TailwindCSS, React Router (PixiJS для Canvas — по плану).
- Общие типы/схемы: пакет `@dnd-table/shared` (Zod).

Совместимость:
- Backend и Frontend взаимодействуют по HTTP (REST) и WebSocket (Socket.IO).
- Все payload’ы и схемы, где возможно, должны соответствовать общим типам/валидациям из `@dnd-table/shared`.

---

### 3.5 Требования к транспортировке и хранению

Развёртывание (локально, для разработки):
- установка зависимостей через `npm` (workspaces);
- конфигурация через `.env`:
  - `infra/backend.env.example` → `backend/.env`
  - `infra/frontend.env.example` → `frontend/.env`

Хранение данных:
- основная БД: MongoDB;
- файлы ассетов: локальная директория `ASSETS_DIR` (MVP).

---

### 3.6 Специальные требования

Интерфейс и UX:
- интерфейс должен быть понятен пользователю средней компьютерной грамотности;
- базовые сценарии должны быть доступны из UI: вход/регистрация, список партий, переход в партию/сцену, выбор объекта, редактирование базовых свойств.

Поэтапная разработка:
- реализация ведётся по вертикалям MVP в приоритете: Auth → Parties → Scene/Objects → Realtime → ACL → Assets;
- модули, разработанные в разное время, должны быть совместимыми (единые схемы, единые ошибки, единая модель прав).

## 5 Технико‑экономическое обоснование

Использование DnDTable снижает трудозатраты на организацию и ведение совместной игровой сессии:
- уменьшает время подготовки сцены (создание/настройка объектов, загрузка ассетов);
- снижает количество ошибок и несогласованности у участников за счёт единого “источника правды” (сервер + синхронизация);
- ускоряет обмен изменениями между участниками (realtime вместо ручной пересылки файлов/описаний).

Ожидаемый эффект:
- повышение эффективности ведения игровой сессии;
- сокращение времени на операции управления сценой и объектами;
- уменьшение вероятности конфликтов данных за счёт версионирования/идемпотентности операций и централизованной проверки прав.
